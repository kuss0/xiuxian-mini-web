# Product Plan

## 一句话定位

为 Telegram 修仙群提供一个轻量 Web 图形副手，把刷屏聊天流整理成重点消息、归档消息和可人工确认的快捷操作，并用 Telegram 官方定时减轻固定 CD 操作。

## 用户问题

- 群消息密度高，玩家聊天、系统公告、bot 回复混在一起。
- 很多玩法依赖复制指令、识别 CD、查看状态，纯聊天窗口操作成本高。
- 固定 CD 玩法可以提前排官方定时，但官方定时无法根据回复内容给出下一步建议。
- 繁体用户对简体命令复制和输入不方便。
- 自动脚本存在封号风险，用户需要更像“客户端辅助界面”的工具，而不是后台自动化傀儡。

## 产品原则

1. 游戏本体仍在 Telegram。
2. Mini Web 只做阅读整理、智能提示、人工确认、官方定时。
3. 所有发送动作必须可解释、可追踪、可取消。
4. 默认不追求极限收益，优先降低异常行为和误触发风险。
5. UI 面向玩家语言，底层保留原始消息和命令。
6. 状态机只辅助展示和排班参考，不驱动自动链式发送。

## 外部游戏 UI 借鉴

参考仓库：

- `JeasonLoop/react-xiuxian-game`：适合借鉴大型修仙 Web 游戏的前端分层。它把界面拆成 `views`、`components`、`hooks`、`services`、`store`、`constants`，每个玩法有独立 view 和 handler，主界面由属性面板、日志流、操作栏、弹窗容器组成。
- `setube/vue-idle-xiuxian`：适合借鉴轻量放置游戏的信息架构。它用顶部功能导航、全局角色摘要、按玩法切页面、Pinia store、IndexedDB、Web Worker 承担离线/批量计算。

Mini Web 不复制它们的游戏本体逻辑，只吸收 UI 和架构方式：

- 第一屏保持“游戏客户端”感：角色摘要、频道/消息流、状态模块、快捷动作同时可见。
- 玩法入口按模块组织：修炼、资源、背包、秘境、副本、风险、设置各自有独立面板，不继续把所有弹层逻辑堆进一个 JS 文件。
- 长列表和历史日志优先服务扫描：主消息流、操作日志、发送日志统一用时间线和过滤器，而不是分散在多个互不关联的弹层。
- 固定规则表进入 `constants` 或后端 domain，不把境界、模块、频道、快捷命令散落在渲染函数里。
- 批量或耗时计算以后放到 worker/后端 job，例如大批消息重算、资源统计重算、库存转移计划；普通 UI 渲染不阻塞。

## 信息分流

底层消息标签：

- `focus`：首页重点流，被 @、会长、关键词、风险和可操作事件。
- `leader`：会长/情报源消息。
- `archive`：点命令、普通 bot 回复和默认不实时关注的消息。
- `world`：世界聊天、普通玩家发言。
- `system`：系统公告、全服事件。
- `mine`：当前角色相关消息。
- `training`：修炼、闭关、元婴、第二元神。
- `dungeon`：副本开启、加入、队伍状态。
- `resource`：储物袋、交易、资源转移。
- `home`：洞府、灵树、小世界、侍妾、法宝、灵兽。
- `risk`：举报、自证、禁言、虚弱、封禁风险。
- `log`：本工具自己的操作记录。

用户视图：

- 重点流：首页默认，只放需要看或可能要人工处理的消息。
- 归档：点命令、普通天尊回复和全量历史，支持按天导出。
- 会长监听：把会长/情报源的消息单独提取出来。
- 操作抽屉：根据消息生成可复制、可确认发送、可加入官方定时的动作卡。

## MVP

第一阶段只做最小闭环：

1. 读取指定 Telegram 群最近消息。
2. 按规则给消息打标签。
3. Web 中展示重点流、会长、风险、副本、资源、归档等快速过滤。
4. 点击消息显示摘要、原文、识别出的角色和玩法。
5. 从消息生成可复制命令。
6. 支持用户确认后发送命令。
7. 支持用命令、间隔/CD、次数创建 Telegram 官方定时消息。

## 暂不做

- 自动挂机调度。
- 自动刷状态。
- 自动补发。
- 自动链式命令。
- 自动答题或防举报 AI。
- 移动端原生 App。
- 内容触发后自动发送下一条命令。

## 后续可扩展

- 自定义解析规则。
- 自定义频道和场景。
- 多账号身份管理。
- 定时排班模板。
- 常用玩法预设，例如深度闭关、抚摸法宝、温养器灵。
- 与现有修仙脚本共享只读状态，但不共享自动发送链路。
